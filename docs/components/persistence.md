# 永続化層 (Persistence)

## 概要
本プロジェクトでは 「Local First」 の思想に基づき、2種類のデータベースを役割分担して使用しています。

- 構造化データには信頼性が高く軽量な SQLite
- ベクトル検索（意味検索） にはAIネイティブな LanceDB

このハイブリッド構成により、シンプルさ・高速性・拡張性のバランスを取っています。

## 構成の選択理由

| 要件               | SQLite                  | LanceDB                     | 選択理由 |
|--------------------|-------------------------|-----------------------------|----------|
| 構造化データ保存   | 非常に得意              | 不得意                      | 解説本文・メタデータ管理に最適 |
| ベクトル類似検索   | 苦手（拡張が必要）      | 非常に得意                  | 将来的なおすすめ機能に必須 |
| 運用負荷           | ほぼゼロ（単一ファイル）| ほぼゼロ（埋め込み型）      | Local LLMプロジェクトに最適 |
| トランザクション   | ACID完全対応            | 基本対応                    | 信頼性確保 |
| 検索性能           | 通常検索は高速          | 意味検索（Semantic Search）が高速 | RAG的な用途に対応 |

## 各DBの役割詳細

### 1. SQLite（構造化データ用）
- 保存内容:
  - 解説本文（`explanations`テーブル）
  - 元のHacker Newsメタデータ（タイトル、URL、投稿日時など）
  - フィードバックデータ（いいね数、評価スコア）
  - Reflection履歴（批評内容、修正前後など）

- 設計ポイント:
  - `PRAGMA journal_mode=WAL;` で同時読み書き性能を向上
  - インデックスを適切に張り、トップページの最新順取得を高速化
  - 将来的にユーザーごとのデータ分離も容易

### 2. LanceDB（ベクトルデータ用）
- 保存内容:
  - 各解説の埋め込みベクトル（Embedding）
  - メタデータ（explanation_id と紐付け）
  - 将来的にマルチモーダルデータ（画像埋め込みなど）も対応可能

- 強み:
  - Apache Arrow / Lanceフォーマットベースで非常に高速
  - 埋め込み型（サーバーレス）で運用が簡単
  - 自動バージョン管理機能あり（データ変更履歴を保持）

## データフロー（保存時）

1. Agent Workflowで最終的な解説が完成
2. Saver Node で以下の処理を実行:
   - SQLiteに構造化データ（本文・メタデータ）を保存
   - テキストをEmbeddingモデルでベクトル化
   - LanceDBにベクトル + explanation_id を保存
3. 両DBで `explanation_id` をFK的に紐付け

## 将来的な拡張計画

- ハイブリッド検索: SQLiteのメタデータ検索 + LanceDBの意味検索を組み合わせ（例: 「量子コンピューティング関連で最近の話題」）
- 類似解説推薦: ベクトル類似度検索で「この解説に似た記事」を表示
- データアーカイブ: 古いデータを圧縮・別ファイル化
- バックアップ戦略: `data/` フォルダ全体をGit LFSや外部ストレージに定期バックアップ
- エクスポート機能: ユーザーが全解説をMarkdownやJSONでエクスポート可能に

## 運用・メンテナンスのベストプラクティス

- データフォルダ: `data/` ディレクトリにすべて集約（.gitignore推奨）
- 定期メンテナンス: VACUUM（SQLite） + LanceDBの最適化
- スキーマ変更: マイグレーションスクリプトを準備（将来的にAlembicなど導入検討）
- パフォーマンス監視: 保存件数が増えた際のクエリ速度を定期的にチェック

## 技術的優位性

この構成は「過剰エンジニアリングを避けつつ、本格的なAI Agentに必要な機能を揃える」というバランスを実現しています。

- ChromaDBやpgvectorのような「全部入り」ソリューションではなく、用途特化の軽量ツールを組み合わせることで、メモリ使用量と起動時間を抑えています。
- Local LLMプロジェクトとして非常に現実的で、再現性の高いアーキテクチャです。
