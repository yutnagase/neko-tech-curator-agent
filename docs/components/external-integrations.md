# 外部連携 (External Integrations)

## 概要
本プロジェクトは 「Local First」 を重視しているため、外部サービスへの依存を必要最小限に抑えています。

現在は Hacker News のみ連携していますが、将来的に複数の技術情報源を追加し、多角的な技術キュレーションを実現する計画です。

## 設計思想

- メタデータ中心: 本文を丸ごと取得せず、タイトル・URL・投稿日時などのメタデータのみ取得
- 著作権・倫理的配慮: コンテンツの複製を避け、LLMによる要約・解説生成をメインとする
- Rate Limiting厳守: 外部サービスに負荷をかけないよう、適切な間隔とキャッシュを徹底
- フォールトトレランス: 外部APIが落ちてもシステム全体が止まらない設計

## 現在の実装

### Hacker News API

- 使用エンドポイント:
  - `/topstories.json`（トップストーリーID一覧）
  - `/item/{id}.json`（個別アイテム情報）

- 取得情報:
  - タイトル
  - URL（外部記事へのリンク）
  - 投稿者（by）
  - 投稿日時（time）
  - スコア（ポイント）

- 実装ファイル: `src/tools/fetchers.py`

- 特徴:
  - 毎日上位30〜50件程度を取得
  - 既に処理済みのURLは重複排除
  - エラー耐性が高く、API変更にも比較的強い

## 将来的に検討・追加予定の外部ソース

| 情報源             | 取得優先度 | 取得内容例                  | 備考 |
|--------------------|------------|-----------------------------|------|
| Reddit (r/MachineLearning, r/programming など) | 高         | 投稿タイトル、URL、Upvote数 | サブレッド指定可能 |

## 技術的考慮事項

### 1. キャッシュ戦略
- 取得したトピックIDをSQLiteにキャッシュ
- 同一日の重複取得を防止
- 将来的に Redis やシンプルなファイルキャッシュも検討

### 2. Rate Limiting & 礼儀正しいクローリング
- リクエスト間隔を最低2〜3秒空ける
- User-Agentにプロジェクト名と連絡先を明記
- 失敗時は指数バックオフでリトライ

### 3. エラー処理
- 外部API障害時は「取得失敗」としてログ出力し、残りの処理を継続
- 部分障害に強い設計（1つのソースが落ちても他は動く）

### 4. データ正規化
- 各ソースから取得したデータを共通の `TopicMetadata` スキーマに変換
- これにより、Agent層はデータソースを意識せずに処理可能

## セキュリティ・プライバシー配慮

- APIキー不要の公開APIのみ使用（現時点）
- 将来的にAPIキーを使う場合は、必ず `.env` で管理し、Gitにコミットしない
- 取得したデータはユーザーのPC内にのみ保存（外部送信なし）

## 今後の拡張方向性

1. プラグイン方式への進化（新しいソースを追加しやすいアーキテクチャ）
2. スコアリング機能（各ソースの信頼度やトピックの重要度を自動評価）
3. マルチソース統合（同じトピックが複数のソースで話題になっている場合に統合表示）
4. ユーザー設定で取得ソースをオンオフ可能に
